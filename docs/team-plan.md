# خطة فريق العمل — assistant404

**دور الإدارة:** تنسيق المهام بين الأعضاء، مراجعة أي تعديل يمس ملفات مشتركة، حل التعارضات بين الأعضاء قبل التنفيذ.

**قاعدة أساسية:** أي عضو يحتاج يعدّل ملف "مشترك" (غير مملوك له حصرياً) يوقف ويبلغ الإدارة أولاً قبل التنفيذ، حتى لو التعديل يبدو بسيط.

---

## ⚠️ قيد معماري يجب أن يعرفه كل الأعضاء

الموقع **Static Site** بدون خادم خلفي (GitHub Pages). أي ميزة "تحكم" أو "تعديل حي" هي أداة تعمل في بيئة التطوير المحلية فقط (`npm run dev`)، والنتيجة تُرفع بـ commit يدوي من صاحب المشروع. لا يوجد تعديل مباشر يراه الزوار فوراً بدون رفع (push) جديد.

## العضو 1 — لوحة التحكم الإدارية (Admin Panel)

### الهدف
بناء واجهة `/admin` تسهّل: إضافة مادة جديدة، إعادة تسمية الأقسام (نظري/عملي/إلخ)، إضافة قسم جديد، إخفاء أو حذف قسم/مادة — بدون الحاجة لتعديل JSON يدوياً بالمحرر.

### الملفات المملوكة (حصرياً لهذا العضو)
```
src/pages/Admin/AdminHome.jsx
src/pages/Admin/AdminSubjectEditor.jsx
src/pages/Admin/AdminSectionsManager.jsx
src/lib/adminStore.js          ← منطق القراءة/الكتابة لملفات JSON محلياً
docs/logs/member-1-admin-log.md
```

### الملفات المشتركة التي سيحتاج لمسّها (تنسيق مسبق مطلوب)
```
src/App.jsx              ← إضافة مسار /admin
src/components/Sidebar.jsx ← إضافة رابط دخول للوحة (يظهر بوضع التطوير فقط)
```

### خطوات التنفيذ المقترحة
1. تصميم شاشة "قائمة كل المواد" مع أزرار: تعديل / إخفاء / حذف
2. شاشة "إضافة مادة جديدة" تولّد مجلد + subject.json + lectures.json فاضي (نموذج)
3. أداة "إدارة الأقسام" تتحكم بـ SECTION_LABELS (تسمية، ترتيب، إخفاء قسم لمادة معينة)
4. آلية الحفظ: نظراً لعدم وجود خادم، تُصدَّر التعديلات كـ **ملف JSON جاهز للنسخ** أو **زر تنزيل** يولّد الملف المحدث ليضعه المستخدم يدوياً في مكانه، ثم يعمل commit
5. توثيق حدود اللوحة بوضوح داخل الواجهة نفسها ("هذه الأداة تعمل محلياً فقط، تحتاج رفع التغييرات يدوياً بعدها")

### حالة "الإخفاء" (Hide) — مهم
حقل موحّد `"hidden": true/false` يُضاف لأي عنصر (مادة، قسم، عنصر محاضرة). كل صفحات العرض (SubjectList, Subject) يجب أن تتحقق من هذا الحقل وتستبعد المخفي — هذا يتطلب تنسيق مع العضو 4 لضمان أن كل نقاط القراءة تراعي هذا الحقل.

## العضو 2 — نظام تعدد الدكاترة/المقررات لنفس المادة (Professor Variants)

### الهدف
نفس المادة (نفس id) قد يكون لها أكثر من دكتور حسب الفصل/الشعبة، وكل دكتور له محتوى مختلف. المدير يقدر يفعّل نسخة دكتور معين ويخفي البقية، بنفس صفحة المادة (بدون إنشاء رابط منفصل).

### الملفات المملوكة
```
src/lib/professorVariants.js   ← منطق اختيار/تصفية النسخة الفعّالة
docs/logs/member-2-professors-log.md
```

### تعديل مخطط البيانات (schema) — يُقترح لكل subject.json
```json
{
  "id": "circuits",
  "name": "الدارات الكهربائية",
  "professorVariants": [
    {
      "professorId": "prof-ahmad",
      "professorName": "د. أحمد",
      "active": true,
      "lecturesFile": "lectures-prof-ahmad.json"
    },
    {
      "professorId": "prof-ali",
      "professorName": "د. علي",
      "active": false,
      "lecturesFile": "lectures-prof-ali.json"
    }
  ]
}
```
كل دكتور له ملف `lectures-{professorId}.json` منفصل بنفس مجلد المادة. الصفحة تعرض فقط النسخة اللي `active: true`.

### الملفات المشتركة التي سيحتاج لمسّها (تنسيق مسبق مطلوب)
```
src/pages/Subject.jsx    ← منطق قراءة النسخة الفعّالة بدل قراءة lectures.json مباشرة
```

### خطوات التنفيذ المقترحة
1. تصميم مخطط `professorVariants` أعلاه والتأكد من توافقه مع لوحة تحكم العضو 1
2. تعديل منطق Subject.jsx ليقرأ الدكتور "الفعّال" فقط
3. حالة توافق عكسي (Backward Compatibility): المواد القديمة بدون `professorVariants` يجب أن تستمر بالعمل بنفس طريقة lectures.json الحالية (لا تكسر شيء موجود)
4. تنسيق مع العضو 1: زر تبديل الدكتور الفعّال يجب أن يكون داخل لوحة التحكم، لا داخل صفحة العرض العامة

## العضو 3 — الوضع المظلم (Dark Mode)

### الهدف
دعم كامل للوضع المظلم مع تباين ألوان مقروء وواضح، وحفظ تفضيل المستخدم.

### المشكلة الحالية (تقنية دين مهمة)
أغلب الكود الحالي مبني على `style={{ color: "#333", background: "#fff", border: "1px solid #ddd" }}` مباشرة بالكود (inline styles ثابتة). هذا **يمنع** الوضع المظلم من العمل تلقائياً، لأن الألوان مكتوبة يدوياً وليست متغيرات.

### الملفات المملوكة
```
src/lib/ThemeContext.jsx     ← إدارة حالة الوضع (فاتح/مظلم) + الحفظ بـ localStorage
src/index.css                ← تعريف متغيرات CSS للألوان (فاتح/مظلم)
docs/logs/member-3-darkmode-log.md
```

### الملفات المشتركة التي سيحتاج لمسّها (تنسيق مسبق مطلوب مع كل الأعضاء)
```
src/App.jsx
src/components/Sidebar.jsx
src/pages/SubjectList.jsx
src/pages/Subject.jsx
src/pages/StudyPlan.jsx
```
**كل صفحة فيها inline styles بألوان ثابتة تحتاج تحويل لمتغيرات CSS.** هذا أكبر مهمة بهذا المسار.

### خطوات التنفيذ المقترحة
1. تعريف متغيرات CSS بـ `index.css`:
   ```css
   :root {
     --bg-color: #ffffff;
     --text-color: #1a1a1a;
     --border-color: #dddddd;
     --accent-color: #2563eb;
   }
   [data-theme="dark"] {
     --bg-color: #121212;
     --text-color: #f0f0f0;
     --border-color: #333333;
     --accent-color: #60a5fa;
   }
   ```
2. بناء `ThemeContext` + زر تبديل بالـ Sidebar
3. المرور على كل صفحة واستبدال الألوان الثابتة بـ `var(--بالمتغير)`
4. اختبار التباين (contrast) يدوياً على كل صفحة بالوضعين

### تنسيق مطلوب
بما إن هذا العضو سيلمس كل الملفات تقريباً، **يجب التنسيق مع الإدارة قبل البدء** لتفادي تعارض مع تعديلات العضو 1 والعضو 2 في نفس الملفات (خصوصاً Subject.jsx).

## العضو 4 — الجودة، التكامل، والتوثيق (QA & Integration)

### الهدف
ضمان أن عمل الأعضاء الثلاثة يعمل معاً بدون تعارض، وتدقيق سلامة البيانات.

### الملفات المملوكة
```
docs/data-schema.md          ← التوثيق الرسمي لأي تغيير بمخطط JSON
docs/id-mapping-audit.md     ← تدقيق تطابق id بين study-plan.json ومجلدات public/pdf
docs/logs/member-4-qa-log.md
```

### المهام الفورية (أولوية عالية)
1. **تدقيق التطابق:** فحص هل كل id في `study-plan.json` يطابق فعلياً اسم مجلد موجود بـ `public/pdf/`. حالياً ظاهر عدم تطابق بين أرقام رسمية (schema سابق) وأسماء وصفية (المجلدات الفعلية مثل `circuits`, `math-1`). يجب توحيد القرار: إما كل المعرّفات slugs، أو كل شي أرقام، ثم تحديث الملفين المرجعيين ليطابقا الواقع
2. مراجعة كل Pull Request/تعديل من الأعضاء الثلاثة قبل الدمج النهائي
3. اختبار الموقع على المتصفح + الجوال بعد كل ميزة تُنجز
4. الحفاظ على قائمة "من يملك أي ملف" محدثة إذا تغيّرت الملكية

## بروتوكول التواصل بين الأعضاء

- أي عضو يحتاج تعديل ملف لا يملكه حصرياً → يوقف وينسق مع الإدارة أولاً
- الإدارة تجمع كل الطلبات المتعارضة وتقرر ترتيب التنفيذ
- كل عضو يحدّث ملف السجل الخاص به (docs/logs/member-X-*.md) بعد **كل** جلسة عمل، وليس فقط بالنهاية

## ترتيب تنفيذ مقترح (لتقليل التعارض)

```
1. العضو 4: تدقيق التطابق أولاً (لأن باقي الأعضاء يعتمدون على مخطط بيانات سليم)
2. العضو 3: بناء نظام الألوان الأساسي (CSS variables) — بدون هذا الأساس، لاحقاً كل تعديل بالألوان يتكرر
3. العضو 1 والعضو 2: يعملان بالتوازي على مسارات منفصلة نسبياً (لوحة التحكم / تعدد الدكاترة)
4. العضو 4: مراجعة ودمج نهائي لكل الميزات معاً
```
